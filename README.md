# RadAI Systems — Food Facility Permit API

## Problem and Solution

The Mobile Food Facilities in San Francisco dataset lists food trucks and carts that hold permits to operate across the city. The goal of this exercise is to build a queryable API over this dataset that answers three questions:

1. **Search by applicant** — given an applicant name, return all matching permit records, with an optional filter by permit status.
2. **Search by address** — given a street address (or prefix), return all permits at matching locations.
3. **Nearest food trucks** — given a latitude and longitude, return the 5 closest permitted food trucks along with their straight-line distance from the provided coordinates.

The solution is a REST API built with FastAPI and backed by a SQLite database. On startup, the raw CSV data is parsed and seeded into the database, making it immediately queryable without any manual setup. A minimal frontend UI is served at `/` for quick manual testing, and full OpenAPI documentation is auto-generated by FastAPI at `/docs`.

---

## Running the Application

```bash
docker build -t radaisystems .
docker run -p 80:80 radaisystems
```

The application will run locally. Navigate to localhost in your browser to send requests, or follow the FastAPI instructions printed to your terminal.

### Setup Explanation

The Dockerfile creates a working directory, copies and installs the dependencies from `requirements.txt`, copies the raw data into the container, navigates to the app directory, and starts the app using the `fastapi` command. On startup, the raw CSV data is seeded into the database. Once seeding is complete, the app is ready to accept requests.

## Repository Structure

```
app/
├── main.py              # App initialization, static route, router registration
├── api/
│   └── permits.py       # Route handlers
├── services/
│   ├── search_by_applicant.py   # Business logic for applicant search
│   ├── search_by_address.py     # Business logic for address search
│   └── nearest_permits.py       # Business logic for nearest-permit lookup
├── models/
│   ├── orm.py           # Permit SQLAlchemy model
│   └── schemas.py       # PermitResponse, NearestPermitResponse
└── db/
    ├── session.py       # Engine, Base, get_db
    └── seed.py          # seed()
```

The source files are organized around separation of concerns. At the root level, `main.py` initializes the app, sets up the database, calls the seed method to populate the SQLite DB from the CSV file in `raw_data/`, and registers the routes defined in `api/permits.py`. There are three routes — one for each required task. Once a request is received, it is handed off to the corresponding service module in `services/`, each of which handles the database access and business logic for one endpoint. Schemas for both database tables and API responses are defined in `models/`.

## Design and Architectural Choices

### Libraries and Packages

I chose to implement the backend portion of this exercise in Python.

**FastAPI and Pydantic** — I chose FastAPI as the web framework because it is the most popular and standard choice in the Python ecosystem. It is easy to work with and provides off-the-shelf OpenAPI documentation for all endpoints.

**SQLAlchemy and SQLite** — For the database, I went with SQLAlchemy and SQLite because they integrate well with FastAPI. I chose a relational database because the food facilities dataset is structured and tabular, which maps naturally to a relational table and makes querying straightforward.

### Business Logic

**Search by applicant and address** — implemented using a SQL query that fetches the matching rows directly.

**Nearest 5 trucks by coordinates** — the brute force approach would be to fetch all rows from the permits table and pick the 5 closest trucks. However, this would load the entire table into memory and is inefficient. Instead, I implemented a bounding box approach to limit the number of rows returned by the SQL query. The bounding box fetches all trucks within a specified distance, and then those results are sorted to find the five closest. If there are no trucks within the initial bounding box, the box is incrementally expanded until it covers enough of the SF region to return at least 5 results.

## What Was Left Out

- **Road distance calculation** — the nearest trucks endpoint returns straight-line (haversine) distance, not actual road distance using the Google Maps API.

- **Dropped columns** — `cnn`, `blocklot`, `block`, `lot`, `X`, `Y`, `NOISent`, `Received`, `PriorPermit`, and `Location` are not stored in the database as they are not needed by any current endpoint.

---

## Alternatives Considered and Trade-offs

### Storage

**In-memory storage** — given the size of this particular dataset, the simplest approach would be to load the entire dataset into memory. However, this quickly becomes inefficient as the dataset grows. Using SQLite means only relevant rows are loaded into memory, and indexes can be added to speed up searches as the data scales. The trade-off is that SQLite has limited concurrency and will need to be replaced as user load increases.

**External database / file mount** — I decided against these in favor of a simple self-contained container that seeds on startup. This makes setup zero-dependency and easy to run locally, but is not production-safe — data does not persist across container restarts and horizontal scaling is not possible. This trade-off is acceptable at this dataset size but would need revisiting for a real deployment.

**Synchronous SQLAlchemy over async** — I used SQLAlchemy in synchronous mode, which is simpler to reason about and requires less boilerplate. The trade-off is throughput under concurrent load, as each request blocks a thread while waiting on the database.

I dropped the following columns from the database as they are not needed for the current endpoints: `cnn`, `blocklot`, `block`, `lot`, `X`, `Y`, `NOISent`, `Received`, `PriorPermit`, `Location`. They can be added back if additional endpoints requiring this data are introduced.

### Nearest Permit Business Logic

I considered using the [Google Maps Distance Matrix API](https://developers.google.com/maps/documentation/distance-matrix/overview) to find the nearest 5 trucks. One advantage is that it calculates actual road distance rather than straight-line displacement, which would be a useful feature as the app gains real users. The main reason I didn't implement it is that it requires a Google Maps API key and each call is charged on a per-request basis. Given the testing overhead this would introduce, I went with the simpler bounding box approach instead — trading distance accuracy for simplicity and zero external dependencies.

### Frontend

Rather than a component-based frontend framework, I used a static HTML page with embedded JavaScript and CSS. This was sufficient to demonstrate the API within scope, but the trade-off is that the frontend is harder to maintain and extend as features grow.

## Security

- **SQL injection** — all queries are constructed using SQLAlchemy's ORM with parameterized bindings, which prevents SQL injection by default.
- **No authentication** — all endpoints are publicly accessible. This is acceptable for a read-only API over public data, but would need to change if write endpoints or sensitive data were introduced.
- **No HTTPS** — the Docker container exposes port 80. In a real deployment, TLS termination should be handled at the load balancer or reverse proxy level.
- **No rate limiting** — the API has no protection against request flooding. Adding rate limiting (e.g. via a middleware or API gateway) would be a priority before exposing this to the public internet.

---

## Problems and Scaling Considerations

**Concurrent request handling** — SQLAlchemy is used in synchronous mode with SQLite. SQLite serializes writes and has limited read concurrency. Under significant user load this would become a bottleneck. Migrating to PostgreSQL  would address both the database and the I/O concurrency constraints.

**Database indexes** — no indexes are defined on the permits table. As the dataset grows, queries on `applicant`, `address`, `latitude`, and `longitude` would benefit significantly from indexes. A spatial index would make the nearest-permit query far more efficient than the current bounding box approach.

**No caching** — repeated identical queries (e.g. nearest trucks for the same coordinates) hit the database every time. Adding a caching layer in front of the service would reduce database pressure substantially for popular queries.

**Seeding on startup** — the database is seeded from the CSV every time the container starts. This is fragile: if the container restarts, any data that diverged from the CSV would be lost or reset. A proper deployment would use a persistent external database and a one-time migration rather than seeding on every boot.

**No horizontal scaling** — the current setup is a single container with a single SQLite file. Scaling horizontally (multiple container instances) would require moving to an external database that all instances share.

**No rate limiting or authentication** — as noted in the security section, these would be necessary before serving real production traffic.
